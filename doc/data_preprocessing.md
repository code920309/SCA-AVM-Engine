# [Portfolio] AppraisalCost AI: 실거래 데이터 전처리 및 통합 파이프라인 구축

본 문서는 AppraisalCost AI 프로젝트의 핵심인 부동산 실거래 데이터의 수집, 정제, 보강 및 시점 수정을 포함하는 전체 전처리 공정을 기록한 문서입니다.

## 1. 프로젝트 개요
*   **목표**: 전국 단위 상업업무용 부동산 데이터를 AI 학습이 가능한 고품질 데이터셋으로 변환
*   **핵심 로직**: 원가법 기반의 감정평가 논리를 데이터 공학적으로 구현
*   **기술 스택**: Python, Pandas, NumPy, Kakao Local API, 공공데이터포털 API

## 2. 데이터 처리 파이프라인 아키텍처

### Phase 1: 원시 데이터 수집 및 기초 정제 (Initial Cleaning)
국토교통부 API로부터 수집된 약 31만 건의 JSON 데이터를 통합하고 신뢰할 수 없는 데이터를 필터링합니다.
- **중복 제거**: 6개 필드(시군구, 법정동, 지번, 거래일, 금액, 면적) 조합의 해시 키를 통한 정밀 중복 제거 (약 15% 감축)
- **노이즈 필터링**: 취소된 거래(`cdealType`), 지분 거래, 특수 목적 건축물(종교, 묘지 등) 제외
- **이상치 처리**: IQR(Interquartile Range) 방식을 활용한 단위 면적당 가격 이상치 제거

### Phase 2: 외부 지수 정규화 (Index Normalization)
시점 보정 계수 산출을 위한 지가변동률 데이터를 정규화합니다.
- **Aggressive Normalization**: 실거래 데이터와 지수 데이터 간의 명칭 불일치를 해결하기 위해 행정구역 접미사(특별시, 도, 시, 구, 군)를 제거한 **'순수 지역 키(Suffix-free Key)'** 생성
- **매핑 성공률 극대화**: "서울특별시 종로구"와 "서울 종로"를 모두 "서울종로"로 변환하여 1:1 매핑 구현

### Phase 3: 유니크 건물 보강 (Enrichment & Optimization)
대규모 데이터셋의 성능 병목을 해결하고 데이터의 밀도를 높입니다.
- **O(N^2) 병목 해결**: 전체 행을 반복 스캔하던 로직을 유니크 건물 데이터프레임 사전 추출 방식으로 개선하여 처리 속도 **1,000배 이상 향상**
- **주소 복원 및 좌표 추출**: 카카오 API를 연동하여 마스킹된 지번을 복원하고 위경도 좌표 및 건축물대장 정보 매핑
- **캐싱 시스템**: `building_cache.json`을 구축하여 API 호출 비용 절감 및 재실행 속도 최적화

### Phase 4: 시점 수정 및 특성 공학 (Time Adjustment & FE)
거시 경제 지표를 개별 거래 데이터에 이식합니다.
- **시점 보정**: 거래 시점의 지수와 기준 시점(현재)의 지수를 비교하여 **보정 계수(Adjustment Factor)** 산출 및 가격 환산
- **Feature Engineering**: 건축물 연식, 층수 정보, 용도별 범주형 인코딩 등 모델 학습용 변수 생성

## 4. 트러블슈팅 (Troubleshooting)

### 이슈 1: 데이터 보강 시 위경도 좌표 결측 발생 (Null Value)
- **현상**: 약 20만 건의 데이터 보강 시도 시 `building_cache.json` 내 좌표값이 100% Null로 생성됨.
- **원인**: 
  - 카카오 `address.json` API의 엄격한 매칭 조건으로 인해 마스킹된 지번(`1**` 등) 처리 시 결과 반환 실패.
  - 건축물대장 API의 사용승인일 정보가 공백(`' '`)인 경우 `int()` 변환 과정에서 `ValueError` 발생 및 프로세스 중단.
- **해결**: 
  - **Hybrid Search 도입**: `keyword.json`(유연한 검색)으로 위치를 먼저 찾고, 이를 통해 얻은 정규 주소로 `address.json`(정밀 검색)을 재시도하는 2단계 검증 로직 구현.
  - **Defensive Parsing**: 모든 수치 변환 로직에 `isdigit()` 및 `try-except` 예외 처리를 적용하여 데이터 정합성 확보.

### 이슈 2: 대규모 데이터셋(16만 건) 처리 병목
- **현상**: 유니크 건물 162,614개 식별 시, 단일 스레드 처리 속도로는 약 200시간 이상의 소요 시간 예상.
- **해결 방안**: 
  - **지역별 우선순위(Tiering)**: 서울, 경기 등 주요 관심 지역 데이터 선제적 구축.
  - **병렬 처리(Concurrency)**: `ThreadPoolExecutor`를 활용한 API 호출 최적화 제안.
